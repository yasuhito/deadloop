# 作業エージェント起動・監視の Cucumber 移行記録

Issue #119 では、Issue 専用の作業場所で担当を重複なく起動し、完了ファイルが確定するまで安全に監視する保証を、日本語の実行可能な受け入れ仕様へ移した。製品動作は変更していない。

## 分類 ID と受け入れ仕様の対応

| 分類 ID | 移行先または扱い | 状態 |
|---|---|---|
| T001 | 既存 pull request の作業場所を開くレビュー担当固有の起動引数、プロンプト書込み、戻り値を一度に診断する低レベル統合テスト | Vitest 継続へ再分類 |
| T002 | 既存 pull request の作業場所を開き、「完了済みの同名レビュー担当は片付けてから交代する」 | 移行済み |
| T003 | 既存 pull request の作業場所を開き、「稼働中の同名レビュー担当は重複起動しない」 | 移行済み |
| T004 | 既存 pull request の作業場所を開き、「複数の同名レビュー担当は片付けない」 | 移行済み |
| T005 | 既存 pull request の作業場所を開き、「別の作業場所にいる同名レビュー担当は片付けない」 | 移行済み |
| T006 | 「準備済み Issue の担当には基準ブランチから専用の作業場所を作る」 | 移行済み |
| T207 | T218 の「調整役が完了ファイルを唯一の完了根拠として監視者へ伝える」低レベルテストへ統合 | Vitest 継続へ再分類 |
| T215 | 「起動した担当を完了ファイルの監視へ引き渡す」 | 移行済み |
| T218 | 調整役が完了ファイルを唯一の完了根拠として監視者へ伝えるプロンプト境界 | Vitest 継続へ再分類 |
| T391 | 「報告要求後に最近活動した担当は監視を続ける」 | 移行済み |
| T392 | 「完了ファイルを求めた後の猶予中は監視を続ける」 | 移行済み |

移行済みの T002〜T006、T215、T391〜T396 は、同じ製品の補助関数または調整役の検証用データを通る受け入れシナリオで置換したため、元の Vitest を削除した。各 Then は一つの外部結果を一つの検査で確認する。

## Vitest を残す境界

次の分類 ID は、受け入れ仕様より失敗箇所を狭く示すため Vitest に残す。

- T001: レビュー担当が既存の作業場所を開く場合の、実行基盤から共通ランチャーまでの低レベル統合。
- T175〜T186: Herdr コマンドの構築、応答の正規化、不正応答の拒否という実行基盤アダプター境界。
- T207、T218: どちらも同じ調整役の検証用データから生成した監視プロンプトを通り、完了ファイルだけを完了根拠として伝える同一の保証だった。重複していた T207 の検査は T218 へ統合して削除し、T218 を残すことで、受け入れシナリオが確認する監視結果より手前の、調整役によるプロンプト生成から監視者への引き渡しまでの文言欠落を直接特定する。
- T216、T217、T219、T222〜T224: 決定論的な名前と完了ファイルのパス、環境変数、描画処理の局所診断。
- T387〜T390: 監視プロンプトの静的文言と定期確認の終了記述の検査。
- T397〜T399: 時刻の解析と CLI 引数エラーの低レベル診断。

これらは公開状態遷移を重複して確認するためではなく、アダプター、文字列、解析、引数のどこが壊れたかを直接示すために残す。

## 同等性と意図的な失敗の確認

2026-07-24 に、削除前の Vitest と新しい Cucumber を併存させて `npm run test:unit` と `npm run test:acceptance` を実行し、両方が成功することを確認した。置換後は Vitest 56ファイル 744テスト、Cucumber 92シナリオ 480ステップが成功した。

同日、この作業場所で Bash の `time npm test` により再計測した。合計 wall time は20.113秒で、Vitest は56ファイル 744テストを14.97秒、Cucumber は95シナリオ 495ステップを4.448秒で完了した。Cucumber 単体は5秒以内だが、合計は10秒以内の目安を超えた。Vitest のクリティカルパスを調べるため `npx vitest run test/enablement.integration.test.ts --reporter=basic` を実行すると、このファイルだけで78テストに13.93秒（コマンドの wall time は14.238秒）かかった。同ファイルは検証用リポジトリを64回、拡張を65回作り直し、各リポジトリ作成で少なくとも7回 git 子プロセスを起動する。したがって、今回追加した Cucumber ではなく、既存の検証用リポジトリと拡張読み込みをテストごとに作り直す境界が主因である。テストを削らずにこの重複コストを減らし、同じ条件で再計測する作業を Issue [#172](https://github.com/yasuhito/deadloop/issues/172) で追跡する。

同日、各移行先の Then の期待結果を一時的に変更して `npm run test:acceptance` を実行した。さらにレビュー修正時に、現行の三つの起動拒否 Then と、T003〜T005 が共有する非起動 Then を一つずつ意図的に壊して再実行した。すべて終了状態1となり、次のとおり実際の Then 名、実際値と一時的な期待値の差、失敗シナリオ数を確認した。

| 分類 ID | 一時的に壊した Then | 報告された差 | 失敗シナリオ数 |
|---|---|---|---:|
| T002 | 完了済みの担当を片付けてから交代を起動する | `remove:finished>create-tab>launch` / `intentional-failure` | 1 |
| T002（起動回数） | 一人の交代担当を起動する（`launchCount`） | `1` / `2` | 1 |
| T003（拒否理由） | 稼働中の同名担当がいるため起動を拒否する | `agent name demo-pr-44-reviewer is working; refusing duplicate launch` / `intentional-failure` | 1 |
| T003（既存担当） | 稼働中の担当は残る | `true` / `false` | 1 |
| T004（拒否理由） | 複数の同名担当がいるため起動を拒否する | `agent name demo-pr-44-reviewer has 2 live candidates; refusing cleanup` / `intentional-failure` | 1 |
| T004（片付け） | 候補を特定できない同名担当は片付けない（`removedAgentIds.length`） | `0` / `1` | 1 |
| T005（拒否理由） | 別の作業場所に同名担当がいるため起動を拒否する | `agent name demo-pr-44-reviewer belongs to a different worktree; refusing cleanup` / `intentional-failure` | 1 |
| T005（片付け） | 別の作業場所の同名担当は片付けない（`removedAgentIds.length`） | `0` / `1` | 1 |
| T003〜T005（非起動） | 新しい担当は起動しない（共有 Then の `launchCount`） | `0` / `1` | 3 |
| T006 | 担当には基準ブランチから Issue 専用の作業場所を作る | `origin/main` / `origin/intentional-failure` | 1 |
| T006（起動回数） | 新しい担当を一人だけ起動する（`launchCount`） | `1` / `2` | 1 |
| T215 | その Issue は完了ファイルの監視対象になる | `worker_monitor_request` / `intentional_failure` | 1 |
| T391、T392 | 担当の監視を続ける（共有 Then） | `continue_waiting` / `intentional_failure` | 2 |
| T393 | 担当に完了ファイルを書くよう求める | `nudge_worker` / `intentional_failure` | 1 |
| T394 | 担当画面の終了を許す | `may_close_pane` / `intentional_failure` | 1 |
| T395 | 終了前に不足した観測を集める | `collect_observations` / `intentional_failure` | 1 |
| T396 | 完了ファイルに従って監視を終える | `promise_settled` / `intentional_failure` | 2 |

T396 はシナリオアウトラインの2例が同じ Then を共有するため、一度の変更で2シナリオが失敗した。2026-07-25 には、別々の保証として追加した二つの起動回数について、各 `launchCount` の期待値を `1` から `2` へ一つずつ変更し、表のとおり対応する1シナリオだけが終了状態1で失敗して `1 !== 2` を報告することを確認した。レビュー修正時には、共有する「新しい担当は起動しない」の期待値を `0` から `1` へ変更した一度の実行で、T003〜T005 の3シナリオだけが失敗することも確認した。各期待値を元に戻した後、`npm run test:acceptance` で 95シナリオ 495ステップが再び成功することを確認した。壊した状態はコミットしていない。
