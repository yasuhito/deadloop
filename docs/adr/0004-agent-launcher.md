# エージェント起動は実行型ランチャーとコード内プロファイル表に一元化する

エージェント種別(ADR 0002)の起動差分は、これまで司令塔プロンプト内の散文分岐として存在していた。起動 argv の組み立ては入力が揃えば完全に決定論的であり、「決定論的に判定できる処理はプロンプトに埋めずスクリプトへ切り出す」(AGENTS.md)の適用漏れだった。第 3 のエージェント CLI(codex, amp 等)の追加を見据え、起動を単一のランチャーコマンドとコード内プロファイル表に一元化する。

## Decision

- **実行型ランチャー** `launch-agent` を追加する。司令塔は Worker / レビューエージェントの起動時にこのコマンドだけを呼ぶ。ランチャーはプロファイルから argv を組み立て、**シェルを介さず**(execFile 相当)実行基盤の起動コマンドを実行し、結果 JSON をそのまま返す。argv を印字して司令塔にシェル展開させる案は、数段落の日本語 prompt を含む引数のクォートを LLM の記述精度に依存させるため退けた。
- **プロファイル表は `src/agent-profiles.ts` のコード内型付きテーブル**とし、エージェント種別の列挙検証(`normalizeProject`)もここから導出する唯一の情報源にする。operator 設定によるプロファイル拡張は認めない(ADR 0002 が自由記述テンプレートを退けたのと同じ理由: エージェント追加は promise 遵守・入力の癖・前提条件の検証を伴う PR 作業)。
- プロファイルは**前提条件**(claude の workspace trust 等)を宣言し、ランチャーが起動前に fail-fast 検査する。未充足が確定 → 起動せず解決コマンド付きエラー。判定不能 → 警告して続行(trust ダイアログは既知の突破手段があるため、不確実な検査で止めるほうが害が大きい)。判定関数は doctor と共有する。
- **実装言語は TypeScript**(node で直接実行、type stripping が使えない環境では素の ESM + JSDoc 型に切替)。core.ts との情報源共有・vitest ネイティブテスト・型検査が理由。あわせて「**新規の決定論ヘルパーは TypeScript を第一言語とし、既存 Python ヘルパーは大きく変更する機会に個別に移行判断**」を方針とする(一括書き直しはしない)。
- スコープは「エージェントを起動する」操作のみ。督促・メッセージング(pane 送信)は別の継ぎ目であり、含めない。将来の実行基盤追加(tmux 等)はこのランチャーに `--runner` 相当を足す形で受ける。

## Consequences

- 司令塔プロンプトからエージェント別の起動コマンド分岐(散文)が消え、テンプレートは実行基盤・エージェント非依存の 1 コマンド呼び出しになる。
- pr-reviewer テンプレートも同じランチャーを使うため、レビューエージェントの claude 対応(#31)は「設定値 `reviewerAgent` を追加して渡すだけ」に縮む。
- エージェント追加の手順が定形化される: プロファイル 1 エントリ + プロトタイプ検証(ADR 0002 の 4 項目)+ テスト、を 1 つの PR で。
