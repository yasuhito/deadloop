# Verify adapters with capability-scoped conformance suites

[アダプター契約テストとCI予算を決める](https://github.com/yasuhito/deadloop/issues/90) で、アダプターの契約適合、常時CIの予算、実環境での対応経路の検証方法を決めた。この判断は、[deadloop coreとアダプターの最小契約](./0006-minimum-adapter-contracts.md)と[第一級対応の受入契約](./0007-first-class-support-contract.md)を、継続的に検証できる形へ具体化する。

## Context

deadloopは、同じ試行の重複起動、古い完了報告、結果不明のGitHub書き込み、dirtyな作業領域の削除などを安全側で扱う必要がある。正常系だけをアダプターごとに個別実装したテストでは、同じ能力を名乗る実装が同じ安全条件を満たすか比較できない。一方、GitHub、Herdr、Workerをすべて接続する試験を通常のPRごとに行うと、認証情報、API制限、外部障害によって常時CIが不安定になる。

判断時点のテストは330件で、手元の実測は`npm test`が約2.3秒、`npm run check`が約3.6秒だった。件数や時間を固定するだけでは、必要な安全試験を削ったり、外部環境の速度差でCIを不安定にしたりする可能性がある。

## Decision

### 1. Reuse conformance suites by capability

契約テストは対応経路全体や製品名ごとではなく、`AttemptJournal`、実行基盤、Agent program、Automation host、目的別GitHub操作などの能力ごとの共通適合試験として表現する。各アダプターは、自身が提供すると宣言した能力の共通試験をすべて通過しなければならない。対応経路は、必要能力を満たすアダプターの組み合わせとして判定する。

共通試験は、内部で呼んだコマンドの回数や順序ではなく、deadloop coreから観測できる結果を検証する。たとえば、同じ試行を2回起動したときに確認するのはHerdr CLIの呼び出し順ではなく、liveなWorkerが1人だけであることとする。

常時CIでは、外部サービスの応答、時刻、通信失敗などを制御可能なテスト用依存へ置き換えるが、アダプター全体をfakeへ置き換えない。製品コードとして使う実際のアダプター実装を、その公開interfaceを通して検証する。

### 2. Test safe failures as permanent invariants

正常系に加え、少なくとも次を共通適合試験の永続的な不変条件とする。

#### Execution runtime

- 同じ試行を繰り返し取得または起動しても、liveなWorkerを二重に作らず、既存資源を復旧する。
- 同じattempt IDに別のIssue、PR、revisionなど異なるbindingが渡された場合は、続行せず契約違反として停止する。
- dirty、未push、所有者不明、状態不明の作業領域を削除せず、場所と復旧手順を示して保全する。
- 停止と後片付けは冪等で、同じ要求を繰り返しても安全な現在状態を返す。

#### Completion report and Agent program

- IssueまたはPR、attempt ID、役割、起動時revisionが一致する有効な完了報告だけを受理する。
- 別の対象、別の試行、古いrevisionの報告を現在の完了証拠として扱わない。
- 必須の検証またはレビュー証拠が欠ける報告を`complete`として扱わない。
- agent固有の追加情報だけを、安全判定やGitHub副作用の根拠にしない。

#### Automation host

- coreが`wait`を返した場合は新しいWorkerを起動しない。
- coreが示した`retry.notBefore`より前に再実行せず、hostが独自の再試行時刻を計算しない。
- coreが`blocked`を返した場合は、理由と復旧手順を提示して自動継続を止める。

#### GitHub operations

GitHub変更要求の送信後に結果が不明になった場合は、同じ変更を直ちに再送しない。現在状態を再取得し、適用済みなら継続、未適用を確認できた場合だけ再送、競合ならワークフローを再評価し、最後まで不明なら安全停止する。コメントは決定的marker、ラベルは期待状態、mergeはexpected head SHAを使って再観測する。

#### Review safety baseline

詳細なレビュー方針は[レビュー方針の最小モデルを決める](https://github.com/yasuhito/deadloop/issues/91)で定めるが、次は先に確定済みの永続的な安全条件として常時CIに残す。

- `autoMerge: false`なら必ず人間へ引き渡し、merge capabilityを使わない。
- レビュー後にPR headが変わった場合はマージしない。
- 必須checkまたは必須approvalを確認できない場合はマージしない。

テストケースは失敗時に壊れた条件を一つに絞れるよう、原則として1ケース1 assertionを維持する。

### 3. Keep CI budgets soft and observable

常時CIには、制御可能な依存だけを使う決定論的な単体試験、共通適合試験、静的検査、型検査、シェル構文検査、パッケージ内容確認を置く。外部サービスを接続する試験は置かない。

件数は330件を基準値とし、固定上限は設けない。約20%増えて400件前後へ達した時点で、重複、実装詳細への過剰な固定、役目を終えた移行テストを見直す。件数を合わせるために永続的な安全試験を削除しない。

時間は手元の標準的な開発環境で、`npm test`を5秒以内、`npm run check`を10秒以内とする。これは見直しを始める予算であり、共有CI runnerの速度差を理由に機械的に失敗させるhard timeoutではない。超えた場合は、安全試験を削る前に、重複の除去、実装の改善、実環境試験の分離を行う。

### 4. Remove tests that preserve history rather than contracts

テストは現在のinterface、利用者向け動作、安全条件、パッケージ契約を守る間だけ残す。過去のファイル名、旧実装、完了済みの移行が存在しないことだけを確認するテストは、その移行期間が終わり、対応コードを削除した時点で削除する。共有契約を破った不具合の回帰試験は永続的な適合試験へ昇格させる。

判断時点では、次の7件を実装段階で再確認する削除候補とする。

- 旧`pi-looper`コマンド名がないこと。
- 旧`PI_LOOPER`環境変数がないこと。
- 改名文書に`breaking cleanup`があること。
- 旧`<promise>`タグがないこと。
- 旧JSONL抽出がないこと。
- 旧`--pane-id`入力がないこと。
- Python自動化のコンパイル確認がCIにないこと。

現在の`deadloop`名、promiseファイルの完了権威、ローカル設定を公開パッケージへ含めないことなど、現在も有効な契約を守るテストは残す。

### 5. Verify live support paths outside normal pull requests

GitHub、実行基盤、Agent programを実際に接続する実環境スモーク試験は、通常のPR CIから分離し、専用のテスト用リポジトリで`autoMerge: false`として実行する。

第一級対応の各対応経路について、少なくとも次の時点で実行する。

- 月1回。
- その対応経路のアダプター変更を含むリリース前。
- 第一級対応へ昇格させる前。

通常の月次確認とリリース前確認では、テスト用Issueからレビュー済みPRを作り、人間へ引き渡して停止する正常系を1回確認する。Automation hostの再起動、同一処理の再実行、重複防止、作業領域の保全を含む完全確認は、第一級対応への昇格前と、アダプターの起動、観測、復旧、後片付けを変更した場合に行う。

このスモーク試験は、作者以外の利用者が公開文書だけで完走する第一級対応の受入確認を置き換えない。

### 6. Keep a public verification ledger

対応経路ごとに専用のGitHub Issueを一つ作り、実環境試験ごとに次をコメントとして記録する。公開文書から最新の合格記録へリンクできるようにする。

- 対応経路と、deadloop、実行基盤、Agent programのバージョン。
- 実行日時と、検証したdeadloop commit。
- テスト用IssueとPRのURL。
- 合否、失敗理由、判定不能の場合はその理由。
- 取得元を明記した、Workerとレビューエージェント別の入力、出力、キャッシュトークン数と試行全体の合計。

トークン数は取得元が確実に報告する場合だけ記録し、取得できない場合は推測せず「取得不可」とする。初期段階では合否条件や上限にせず、最適化の基準値として扱う。この記録はdeadloopの公式検証に限定し、試験利用者から自動収集するテレメトリーにはしない。

実環境試験がGitHub障害やテスト用認証情報の問題で製品を確認できなかった場合は「判定不能」と記録し、環境を直して再実行する。deadloopまたは対応経路が重複防止、成果保全、安全停止などの受入契約に違反した場合は、再び合格するまでその対応経路を実験的対応へ戻す。

## Consequences

- 新しい実行基盤やAgent programは、製品名別の独自テストではなく、宣言した能力の共通適合試験を通して追加する。
- 常時CIは高速で決定論的な安全条件へ集中し、外部サービスの不安定さを通常のPRへ持ち込まない。
- 第一級対応は一度の成功ではなく、月次と変更時の実環境証拠によって継続的に確認される。
- トークン数を対応経路別に比較できるが、利用者の同意のない自動テレメトリーは導入しない。
- 実装段階では、共通適合試験の入口を先に作り、既存テストを移すたびに重複と履歴固定テストを削除する。
