# Qualify support paths by an observable acceptance contract

[推奨経路と第一級対応の受入契約を確定する](https://github.com/yasuhito/deadloop/issues/89) で、90日内に利用者へ推奨する第一級対応と、実験的対応の境界を決めた。製品名単体ではなく、Automation host、実行基盤、Agent programの組み合わせを一つの対応経路として判定する。

## Decision

90日内の基準経路は、Pi Automation hostとHerdr実行基盤の組み合わせとする。Pi Workerを基準にし、Claude Code WorkerとCodex Workerは、同じ受入契約を満たしたものから個別に第一級対応へ昇格させる。三種類すべての昇格をあらかじめ約束しない。

Claude App内またはCodex App内の定期実行だけでdeadloopを動かす経路は、90日内は実験的対応とする。Appが管理する定期実行、作業領域、セッションをdeadloopが外部から一貫して再発見、再開、回収できる公開契約を確認できていないためである。

対応経路を第一級対応とするには、少なくとも次の観測可能な結果をすべて満たす。

- 公開手順に従って導入し、GitHub Issueからレビュー済みPRを作って人間へ引き渡せる。自動マージは第一級対応の必須条件にしない。
- 必須能力と設定を副作用より前に確認し、未対応機能を推測的な代替処理で隠さない。
- Automation hostの再起動や同じ処理の再実行後も既存の試行を再発見し、Worker、PR、コメントなどの副作用を重複させない。現在状態を証明できなければ続行しない。
- 自動継続できない場合は成果を保全して停止し、停止理由、保全した作業場所、確認または復旧の手順を示す。すべての失敗を自動復旧する必要はない。
- 後片付けでは試行の所有物で安全に削除できることを確認し、dirty、未push、競合、所有者不明、状態不明の作業領域を削除しない。
- その対応経路について、作者以外の利用者が少なくとも1人、新しいテスト用リポジトリで公開文書だけを使い、導入からレビュー済みPRまで完走する。口頭の手順補足が必要になった場合は未合格とし、文書を修正して再確認する。

受入契約を一つでも満たしていない、または確認していない対応経路は実験的対応と表示する。公開文書には未確認能力と既知の制約を記載し、実験的対応では`autoMerge: true`を許可しない。

## Consequences

第一級対応は「CLIを起動できた」「一度PRができた」といった実装状況ではなく、経路ごとの証拠で昇格させる。Pi + HerdrでAgent programを追加する場合も、Claude AppまたはCodex AppをAutomation hostとして追加する場合も、同じ境界を使う。

常時CIへ置く契約テスト、実環境スモークテスト、時間と件数の予算は、[アダプター契約テストとCI予算を決める](https://github.com/yasuhito/deadloop/issues/90)でこの受入契約から導出する。READMEと初回導入手順は、[READMEと初回導入体験の優先改善を決める](https://github.com/yasuhito/deadloop/issues/92)で第三者確認を実行できる形へ具体化する。
