# PROTOTYPE — 第三者導入観察票

> **使い捨ての試作品です。** Issue「[Pi + Herdrの第三者導入で最初の障壁を観察する](https://github.com/yasuhito/deadloop/issues/86)」の観察方法を具体化し、2〜3人で試した後に削除または正式な試験利用手順へ吸収します。

## Wayfindingでの判断

2026-07-10時点では、身近にPi + Herdrを利用している第三者がおらず、ロードマップ策定前の観察は実施しない。自然発見による報告を無期限に待つのではなく、試験利用者の募集と観察を90日ロードマップの初期実行項目へ移す。この観察票は参加者を募集できた時点で使用する。

## この試作品が答える問い

作者以外の利用者が、公開資料だけを頼りに Pi + Herdr 経路を導入して、テスト用 Issue から `ready-for-human` の PR まで安全に到達できるか。到達できない場合、最初に共通改善へ昇格すべき障壁は何か。

## 参加者

次を満たす2〜3人を対象にする。

- 作者ではない。
- 自分で管理する GitHub リポジトリか、試験用リポジトリを使える。
- Pi と Herdr を使った経験の有無は問わない。ただし経験の有無を記録する。
- 観察中に共有する情報と、公開してよい情報を自分で選べる。

同じ社内環境だけに偏らないことが望ましいが、最初の2〜3人の募集を遅らせる条件にはしない。

## 安全条件

- `autoMerge: false` を固定する。
- 機密情報を含まない、破棄可能なリポジトリまたは小さなテスト用 Issue を使う。
- 参加者が画面共有、記録、公開のいずれかを断っても試験参加を続けられる。
- API キー、トークン、ローカルパス、非公開リポジトリ名は観察票へ転記しない。
- 作業領域に未コミット差分が残った場合、CI・レビュー・権限確認が失敗した場合、または意図しない GitHub 副作用が起きそうな場合は止める。回避して先へ進まない。

## 観察前に参加者へ伝える文

> deadloop の初回導入手順を試して、文書だけで GitHub Issue からレビュー済み PR の人間確認待ちまで到達できるかを確認します。製品ではなく手順を評価するので、迷ったらそのまま声に出してください。私は原則として答えを教えず、公開資料から自己解決できるかを観察します。安全上の問題があるときだけ止めます。共有する記録と、後で公開してよい内容は最後に個別に確認します。

## シナリオ

参加者にはリポジトリの [README](../../README.md) から開始してもらい、次のゴールだけを伝える。

> 自分の試験用リポジトリで deadloop を安全側の設定で起動し、小さな Issue に適格ラベルを付け、Worker の実装、検証、PR 作成、レビューを経て `ready-for-human` で止めてください。マージはしないでください。

手順一覧は事前に渡さない。観察者だけが次の段階を採点に使う。

1. 対象利用者と権限の理解
2. deadloop の導入と Pi 拡張の有効化
3. Pi / Herdr / `gh` の前提確認
4. `deadloop.json` または安全なローカル上書きの設定
5. 必須ラベルの作成
6. 対象リポジトリから Pi を起動
7. `/deadloop-doctor` による診断
8. テスト用 Issue の作成と適格ラベルの付与
9. Worker の起動、実装、検証、PR 作成
10. レビューと `ready-for-human` への引き渡し
11. 停止理由と次の人間操作の説明

## 観察の進め方

- 開始時刻を記録する。
- 参加者には「考えていることを声に出す」よう依頼する。
- 迷い始めた時刻、最初に探した場所、実行した確認方法を記録する。
- 10分間は安全上の問題を除いて助言しない。
- 参加者が助けを求めたら、まず「次に何を調べようとしていますか」と一度だけ聞く。
- それでも進めない場合は最小のヒントを一つだけ渡し、内容と時刻を記録する。
- コマンドの貼り付けミスは製品障壁とせず、資料から誤ったコマンドを選んだ場合は製品障壁候補にする。
- `ready-for-human` 到達時、または安全停止・参加者辞退・60分経過時に終了する。

障壁の優先順位は次の順で決める。

1. 一度でも危険側へ進ませた障壁
2. 二人以上を同じ段階で止めた障壁
3. 一人でも自己解決不能にした障壁
4. 二人以上へ同種のヒントが必要だった障壁
